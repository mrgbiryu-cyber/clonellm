## 1. 현재 시스템 본질

> **레퍼런스 웹사이트를 슬롯 단위로 분석하고, LLM 파이프라인(Planner→Composer→Builder→Critic)을 통해 섹션별 리디자인을 자동 생성·비교·미리보기하는 AI 디자인 런타임**

핵심 파이프라인:
```
POST /api/build
  → Planner  : 요구사항 → 컨셉 계획
  → Composer : 구조 의도 설계
  → Builder  : 디자인 오퍼레이션 생성
  → Fixer    : visual critic 실패 시 수정
  → Critic   : 시각 품질 평가
```

LLM 5개 역할 모두 독립 모델 설정 가능 (`PLANNER_MODEL`, `COMPOSER_MODEL` 등).

## 2. 에이전시 활용 시나리오

에이전시가 가장 직접적으로 쓸 수 있는 3가지:

| 시나리오 | 설명 | 현재 가능 여부 |
|---|---|---|
| 제안 단계 시안 자동화 | 클라이언트 사이트 clone → 방향별 시안 3~5종 생성 | 부분 가능 (LGE 한정) |
| 캠페인 시즌 변형 | 봄/여름/추석/연말 홈 변형 정기 생성 | 부분 가능 |
| A/B 테스트 소재 생산 | hero/quickmenu 변형 다수 생성 후 비교 | 가능 |

에이전시 도구로 확장하려면 **범용 URL 지원**이 전제 조건. 현재는 LGE 사이트 구조가 하드코딩되어 있다.

## 3. 획기적 고도화 방향 (구독형 서비스 관점)

구독 가치가 성립하려면: 반복성 + 대체 인건비가 커야 한다.

### 3-1. 경쟁사 디자인 인텔리전스

```
경쟁사 사이트 정기 크롤
→ 슬롯 단위 변화 감지
→ 변화 의미 해석
→ 우리 사이트 대응 시안 자동 생성
```

- **차별점**: Visualping/Crayon은 텍스트/메시지 변화 추적. 디자인 구조 레벨 감지+대응 시안 생성은 시장에 없음
- **구독 이유**: 경쟁사는 매주 바뀐다

### 3-2. 자율 디자인 최적화 루프 (가장 획기적)

```
[LLM 변형 생성] → [실 트래픽 일부 배포] → [전환율 수집 2주]
       ↑                                              ↓
[다음 세대 변형]  ←  [승자 패턴 학습]  ←  [결과 분석]
```

- **현재 시장 갭**: A/B 테스트 툴은 디자인 생성 불가. AI 디자인 툴은 성과 데이터 연동 없음
- **Leadpages** 가 유사한 방향 ($399/월)이지만 텍스트/CTA 레벨. 구조적 디자인 변형은 없음

### 3-3. 시즌/캠페인 디자인 캘린더 자동화

```
[연간 캠페인 캘린더] + [브랜드 가이드라인]
→ 4주 전 자동 시안 5종 생성
→ 담당자 승인 버튼만
→ 다음 해에는 전년도 성과 기반으로 더 정교하게
```

- **대상**: 대형 이커머스/유통사 (설/봄/추석/블프/크리스마스 반복)

### 우선순위

| 순위 | 방향 | 이유 |
|---|---|---|
| 1 | 자율 최적화 루프 | 시장에 없는 카테고리. ROI가 전환율로 직접 측정 |
| 2 | 경쟁사 인텔리전스 | 기업이 이미 돈 쓰는 영역. 대응 시안 자동화가 차별점 |
| 3 | 시즌 캘린더 자동화 | 이커머스에서 수요 명확 |

## 4. 여정 설계 (Journey Design) 개념

### 4-1. 패러다임 전환

| 관점 | 현재 (페이지 단위) | 여정 설계 |
|---|---|---|
| 작업 단위 | 페이지 1개 | 콘셉트 패키지 1개 |
| 사이드바 | 페이지 목록 | 여정 맵 + 페이지 목록 |
| 요구사항 | 기술 파라미터 중심 | 방향 텍스트 중심 |
| 결과 확인 | 페이지별 비교 | 여정 전체 프리뷰 |
| 일관성 관리 | 없음 | 페이지 간 DNA 일치율 표시 |

### 4-2. 콘셉트 패키지 (핵심 개념)

홈에서 정의되고 여정 전체를 관통하는 디자인 DNA 묶음.

```json
{
  "conceptId": "editorial-cinematic-v2",
  "createdFrom": "home",
  "dna": {
    "tone": "cinematic",
    "surfaceRhythm": "dark-with-light-accent",
    "hierarchy": "visual-first",
    "density": "spacious",
    "imageRole": "immersive-dominant"
  },
  "journeyPages": ["home", "category-tvs", "category-refrigerators", "clone-product"]
}
```

현재 `designIntent` + `tone`이 워크스페이스에 페이지별로 저장된다. 이 위에 패키지 레이어를 얹으면 된다.

### 4-3. 역할별 DNA 적용

같은 DNA라도 각 페이지 역할에 따라 적용 강도가 다르다.

| 페이지 | 역할 | DNA 적용 방식 |
|---|---|---|
| 홈 | 브랜드 세계관 진입 | 전면적, hero + quickmenu 모두 |
| 카테고리 | 탐색 + 발견 | DNA 배경 유지, 정보 격자 구조 우선 |
| PDP | 구매 설득 | DNA는 배경색/타이포로만, 제품 집중 |
| 장바구니 | 전환 결정 | DNA 최소화, 신뢰/명료성 우선 |
| 결제 | 완료 | 브랜드 일관성 유지, 불안 제거 |

### 4-4. Admin UI 수정 방향

**사이드바**: 여정 맵 탭 추가
```
[페이지 목록] [여정 맵]  ← 탭 전환

여정 맵:
  홈
   ↓
  카테고리 (TV / 냉장고)
   ↓
  PDP → 장바구니 → 결제
```

**요구사항 섹션**: 기술 파라미터 숨김
```
현재: mode / patchDepth / rendererSurface / builderProvider 노출
변경: "이번에 원하는 방향" 텍스트 + "전체 여정 적용" 체크박스
      [고급 설정 펼치기] → 기존 파라미터
```

**새 섹션 추가**: 여정 콘셉트
```
현재 위치: 홈 → [카테고리] → PDP → 장바구니 → 결제
활성 콘셉트: "Editorial Cinematic v2"
  tone: cinematic / density: spacious / hierarchy: visual-first
[이 콘셉트로 빌드] [전체 여정 적용]
```

**aside 확장**: 여정 프리뷰 + 일관성 점수
```
여정 프리뷰 [홈→카테고리→PDP 순서로 보기]

여정 일관성
  홈        ████████ 일치
  카테고리   ████░░░░ 70%
  PDP       ██░░░░░░ 42% ⚠
  장바구니   ░░░░░░░░ 미구현
  결제       ░░░░░░░░ 미구현
```

## 6. 전환 구간 (장바구니/결제) 문제

### 6-1. 왜 현재 커버 안 되는가

1. **로그인 필요** — 세션 없이 접근 불가
2. **결제 게이트웨이** — 외부 PG사 페이지로 이동
3. **트랜잭션 상태 의존** — 장바구니에 담긴 상품이 있어야 보임
4. **sitemap 미등록** — 크롤러가 접근할 URL이 없음

### 6-2. 시스템이 이미 알고 있는 것

`server.js:3414`에 이미 매핑 존재:
```js
cart: "cart",
checkout: "checkout",
pdp_to_cart: "cart",
cart_to_checkout: "checkout",
purchase_flow: "checkout"
```

레퍼런스 데이터도 이미 수집됨:
- **Baymard**: 18,000+ 이커머스 UX 패턴, cart/checkout 포함 ✓
- **Mobbin**: cart/checkout 포함 ✓
- **Pageflows**: `pdp_to_cart`, `cart_to_checkout` 전용 (현재 403 오류)

### 6-3. 선택된 접근법: Reference 기반 전환 구간 (Option B)

live clone이 아닌 Baymard/Mobbin 레퍼런스로 cart/checkout 패턴을 LGE 브랜드에 맞게 설계.

근거:
- Baymard 데이터 이미 `fetchStatus: ok`
- LGE 브랜드 DNA(색상, 타이포, 공간감)를 적용해서 "LGE가 만들었을 형태"로 생성 가능
- 여정 흐름 설계 목적에는 충분

### 6-4. 전환 구간 타입별 차이

| 여정 유형 | 전환 구간 구조 |
|---|---|
| 일반 구매 | 장바구니 → 주문서 작성 → 결제 → 주문완료 |
| 케어솔루션 구독 | 구독 신청 폼 → 플랜 선택 → 결제 → 가입완료/마이페이지 |
| 베스트샵 | 온라인 견적/예약 → 매장 방문 → 오프라인 구매 |

| 문제 | 상세 |
|---|---|
| 페이지 간 맥락 없음 | 카테고리 작업 중 홈과의 관계가 안 보임 |
| 기술 파라미터 우선 노출 | 첫 입력이 mode/patchDepth/rendererSurface — 의도보다 설정이 먼저 |
| 콘셉트 연결 없음 | 홈에서 선택한 방향이 카테고리에 이어지지 않음 |
| 결과 확인 단절 | 빌더 결과를 한 페이지씩만 볼 수 있음 |
| aside 정보 산만 | 자산/이력/버전이 분리된 details로 흩어져 있음 |

### 9-9. 구현 우선순위 (UI)

| 순위 | 항목 | 현재 코드 영향 범위 | 난이도 |
|---|---|---|---|
| 1 | 요구사항 섹션 UX 단순화 | requirements 섹션 HTML + JS 리팩토링 | 낮음 |
| 2 | 사이드바 탭 구조 (페이지목록 탭 분리) | sidebar HTML + JS 탭 전환 로직 | 낮음 |

| 4 | 여정 콘셉트 섹션 신규 | 새 section + 콘셉트 패키지 데이터 구조 | 중간 |
| 5 | aside 여정 프리뷰 패널 | aside HTML + 일관성 점수 로직 | 중간 |
| 6 | 사이드바 여정 맵 탭 | 노드 그래프 렌더링 + 여정 정의 데이터 | 높음 |

**1, 2번부터 시작**: 기존 기능을 해치지 않으면서 즉시 UX가 개선된다.
**3, 4, 5번**: 콘셉트 패키지 데이터 구조가 정의되면 순차 진행.
