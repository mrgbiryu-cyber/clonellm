# 관리자 연구용 화면 정리 및 서버 반영 가이드

## 1. 목적

이 문서는 `web/admin-research.html`에 반영한 연구용 관리자 화면 변경을 정리하고,

- 어떤 구조/스타일을 실험했는지
- 현재 무엇이 반영되었는지
- 운영 서버 화면에 반영할 때 무엇을 옮겨야 하는지

를 한 번에 확인하기 위한 문서다.

현재 화면은 **운영 적용본이 아니라 연구용 프로토타입**이다.

즉:

- 운영 `web/admin.html`은 아직 직접 수정하지 않음
- 연구용 화면에서 구조와 UI를 먼저 확인
- 괜찮다고 판단되면 운영 화면으로 이식

하는 흐름을 전제로 한다.

## 3. 이번 연구 범위

이번 연구는 전체 관리자 기능을 다루지 않고 아래 3개를 우선 닫는 구조를 목표로 했다.

1. 페이지 정체성
2. 요구사항
3. 기획서

그리고 기획서 이후 연결되는 다음 단계로:

4. 디자인 빌더

까지 문서 흐름 안에서 이어지는지 확인하는 구조를 실험했다.

빌더 실행 결과의 실제 미리보기/비교/버전 반영은 현재 운영 시스템의 로직을 그대로 쓰는 것이 아니라,
연구용 UI 흐름에서 어디에 둬야 자연스러운지를 확인하는 목적이다.

## 4. 화면 구조 요약

## 4.1 좌측

좌측은 페이지 선택 전용으로 유지했다.

- 검색
- 상태 필터
- 그룹형 페이지 목록

목록 그룹:

- 홈
- 서비스
- PLP
- PDP

좌측 항목은 최신 `admin.html`의 실제 타깃 구조를 기준으로 맞췄다.

즉 현재 연구용 화면의 목록은 샘플 3개가 아니라,
운영 화면의 `adminTargetCatalog` 기준 구조를 반영한 상태다.

## 4.2 우측 상단

- 페이지명
- 페이지 유형
- 최근 편집자

그리고 본문 진입 전:

- `페이지 정체성`
- `요구사항`
- `기획서`

상태 요약 바를 유지했다.

## 4.3 우측 본문

본문은 4개 섹션으로 정리했다.

1. 페이지 정체성
2. 요구사항
3. 기획서
4. 디자인 빌더

기본 상태에서는 `디자인 빌더`는 숨겨져 있고,
기획서에서 `빌더 승인 및 실행`을 누르면:

- 앵커에 `디자인 빌더`가 추가되고
- 본문 하단에 `디자인 빌더` 섹션이 나타난다.

## 4.4 우측 보조 영역

보조 영역은 아래 순서로 정리했다.

1. 관련 화면
2. 버전 확인
3. 접어두는 보조 기능
4. 운영 전용 도구

이 분리는 기능 매핑표 기준으로 정한 것이다.

## 5. UI/CSS 변경 요약

## 5.1 전체 레이아웃

- 좌측 사이드바 + 우측 상세 구조 유지
- 우측은 `상태 요약 바 -> sticky anchor -> 본문 + 보조 영역` 구조로 정리
- 본문 섹션 카드와 보조 영역을 역할별로 분리

## 5.2 sticky anchor

앵커:

- 페이지 정체성
- 요구사항
- 기획서
- 디자인 빌더(조건부)

각 섹션 카드에 `scroll-margin-top`을 넣어,
앵커 클릭 시 상단에 제목이 가려지지 않도록 조정했다.

## 5.3 설명문 정리

기존처럼 제목 아래에 긴 설명문이 항상 보이면 화면 밀도가 높아지는 문제가 있어,

- 페이지 헤더
- 페이지 정체성
- 요구사항
- 기획서
- 디자인 빌더
- 관련 화면

의 설명을 제목 오른쪽 `i` 아이콘 툴팁으로 이동했다.

즉 설명은 상시 노출이 아니라:

- hover
- focus

시 확인하는 방식으로 정리했다.

## 5.4 관련 화면

관련 화면은 길게 설명하지 않고,
명칭 자체가 역할을 드러내게 바꿨다.

현재 항목:

- 작업 미리보기
- 전후 비교
- 원본 홈페이지 디자인 / 원본 기준 화면

버튼명은 모두 `열기`로 통일했다.

## 5.5 버전 확인

버전은 관련 화면과 분리했다.

이유:

- 버전은 누적 자산이라 목록이 길어질 수 있음
- 관련 화면은 즉시 이동용이어야 함

따라서:

- 관련 화면 = 즉시 열기
- 버전 확인 = 접힘 섹션에서 관리

로 역할을 분리했다.

## 5.6 요구사항 입력 방식

운영 화면의 planner 입력 방식을 참고해 연구용 화면에도 아래를 넣었다.

- 입력 방식
  - 직접 작성 + 레퍼런스
  - 직접 작성
  - 레퍼런스 사이트 입력
- 디자인 변화율
  - 하 / 중 / 상

그리고 입력 방식에 따라 필드를 다르게 보이게 했다.

현재 규칙:

- `hybrid`
  - 레퍼런스 URL
  - 원하는 방향
  - 톤앤매너
  - 요청 제목
  - 핵심 메시지
  - 피해야 할 방향
  - 요청 배경
- `direct`
  - 요청 제목
  - 핵심 메시지
  - 피해야 할 방향
  - 요청 배경
- `reference`
  - 레퍼런스 URL
  - 원하는 방향
  - 피해야 할 방향

## 5.7 디자인 빌더 섹션

기획서 아래 생성되는 빌더 영역은
전체 페이지 iframe이 아니라
**섹션별 결과 카드** 구조로 정리했다.

현재 카드 구조:

- 섹션명
- 슬롯명
- 변경 내용
- 비교 열기
- 자세히 보기

처음엔 카드 안에 많은 정보가 있었지만,
현재는 `변경`만 남겨서 더 간결하게 줄였다.

카드 레이아웃은 `1열`로 정리했다.

## 5.8 버전 액션

버전 확인 영역의 각 버전에는 아래 액션을 두었다.

- 클론에 적용
- 이 버전 보기
- 비교 열기

즉 현재 연구용 흐름에서는
버전은 단순 열람 목록이 아니라
실제 반영 단위로 가정하고 있다.

## 6. 현재 화면에서 확인 가능한 흐름

현재 연구용 화면에서 가능한 흐름:

1. 왼쪽에서 페이지 선택
2. 상단 상태 요약으로 진행 상태 확인
3. 페이지 정체성 검토
4. 요구사항 입력
5. 기획서 확인
6. 기획서의 `빌더 승인 및 실행`
7. 디자인 빌더 섹션 노출
8. 섹션별 결과 카드 확인
9. 관련 화면 열기
10. 버전 확인 및 클론 적용

---

## 7. 현재 상태에서 의도적으로 남겨둔 것

이 화면은 연구용이므로 아래는 실제 동작보다 구조 표현이 우선이다.

- 관련 화면의 `열기` 버튼은 실제 URL 연결 없음
- 버전 확인의 `클론에 적용`은 실제 API 연결 없음
- 디자인 빌더 섹션의 결과 카드는 실제 렌더 이미지가 아니라 와이어 형태
- 일부 상세 데이터는 샘플 상태 데이터 기반

즉 **구조/동선 검증용**이지,
실제 서버 동작 연결까지 끝낸 화면은 아니다.

## 8. 운영 서버 반영 시 이식 단위

운영 화면에 적용할 때는 한 번에 전부 합치지 말고 아래 단위로 옮기는 것이 안전하다.

## 8.1 1단계

`admin.html`에 아래만 우선 이식

- 좌측 페이지 목록 구조 재정리
- 우측 상세 상단 상태 요약 바
- sticky anchor
- `정체성 / 요구사항 / 기획서` 3섹션

## 8.2 2단계

요구사항 입력 방식 반영

- 입력 방식 셀렉트
- 디자인 변화율 셀렉트
- mode에 따른 필드 show/hide

## 8.3 3단계

기획서 하단 빌더 연결 반영

- `빌더 승인 및 실행`
- 조건부 `디자인 빌더` 섹션

## 8.4 4단계

우측 보조 영역 이식

- 관련 화면
- 버전 확인
- 접어두는 보조 기능
- 운영 전용 도구

## 8.5 5단계

운영 로직 연결

- 관련 화면 버튼 -> 실제 URL
- 버전 확인 -> 실제 저장 버전 데이터
- `클론에 적용` -> 실제 pin/apply 로직

## 9. 서버 반영 시 주의

1. `admin-research.html`은 현재 독립 정적 화면이다.
   운영 반영 시 그대로 rename 하지 말고, `admin.html` 구조에 부분 이식해야 한다.

2. 운영 화면은 실제 데이터/이벤트가 이미 많으므로,
   구조를 먼저 이식하고 기존 이벤트를 붙이는 순서가 안전하다.

3. 특히 아래는 운영 로직과 강하게 연결돼 있으므로
   UI만 복사하면 안 된다.

- 입력 방식별 planner payload
- 기획서 저장
- 빌더 실행
- 저장 버전 목록
- 현재 View/클론 반영

4. `관련 화면`은 실제 URL 빌더 함수와 연결해야 한다.

운영 쪽 기준 후보:

- preview
- compare
- page compare
- clone

5. `버전 확인`은 실제 `savedVersions`와 `pinned view` 데이터를 연결해야 한다.

## 10. 권장 반영 순서

운영 서버 적용은 아래 순서가 가장 안전하다.

1. 구조 이식
2. 상태 요약 연결
3. 요구사항 mode 로직 연결
4. 기획서 영역 이식
5. 빌더 승인/실행 연결
6. 관련 화면 연결
7. 버전 확인 연결
8. 운영 전용 도구를 하단 격리

즉,

**레이아웃 -> 상태 -> 입력 -> 기획서 -> 빌더 -> 버전**

순으로 가는 것이 좋다.
