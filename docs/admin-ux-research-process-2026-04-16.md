# 관리자 화면 개편 연구 진행 방식

## 원칙

이번 작업은 바로 운영 화면에 반영하지 않는다.

먼저 아래 순서로 진행한다.

1. **연구용 화면**을 별도로 만든다.
2. 현재 운영 `admin.html`은 건드리지 않거나, 최소한으로만 유지한다.
3. 연구용 화면을 로컬에서 직접 보면서 수정한다.
4. 구조와 사용성이 괜찮다고 판단되면 그때 운영 화면에 이식한다.

즉 이번 단계의 목표는 “적용”이 아니라 **검토 가능한 연구용 프로토타입 제작**이다.

---

## 진행 방식

### 1. 운영 화면과 분리

- 현재 운영 파일: `web/admin.html`
- 연구용 프로토타입: 별도 파일로 생성

이렇게 분리하는 이유는:

- 운영 화면을 흔들지 않기 위해
- 과감한 구조 실험을 하기 위해
- 승인 전까지 dead path를 만들지 않기 위해

### 2. 연구용 화면 범위

이번 연구용 프로토타입은 아래 3개만 대상으로 한다.

1. 페이지 정체성
2. 요구사항
3. 기획서

빌더, 비교, 버전, runtime, acceptance는 이번 연구 화면에서 메인 대상이 아니다.

### 3. 검토 방식

연구용 화면을 만든 뒤:

1. 로컬에서 직접 연다
2. 사용자와 같이 화면을 보며 피드백을 반영한다
3. 구조, 위계, 내비, 섹션 구성을 반복 수정한다
4. 괜찮다고 판단되면 실제 `admin.html`에 적용한다

---

## 구현 원칙

### 1. 연구용 화면은 독립 HTML로 만든다

가능한 한 운영 코드와 분리된 단일 파일로 만든다.

장점:

- 빠른 수정
- 서버 로직 영향 없음
- 비교가 쉬움

### 2. 실제 데이터 구조는 참고하되, 화면 실험이 우선이다

연구 단계에서는

- 현재 admin의 실제 필드 구조
- 실제 섹션 이름
- 실제 문서 흐름

은 유지하되,

레이아웃과 정보 구조는 자유롭게 재조합한다.

### 3. 승인 전까지는 운영 라우트에 연결하지 않는다

즉 연구용 화면은:

- `/admin` 대체 금지
- 운영 메뉴에 연결 금지
- 문서/파일 경로로만 확인

---

## 최종 적용 기준

아래가 만족되면 운영 화면으로 반영한다.

1. 좌측 목록 구조가 명확하다
2. 우측 상세가 `정체성 / 요구사항 / 기획서` 중심으로 읽힌다
3. sticky anchor 포함 내부 이동이 자연스럽다
4. 문서형 흐름이 관리자 화면 안에서 설득력 있게 보인다
5. 빌더/운영 메타가 전면을 방해하지 않는다

---

## 결론

이번 관리자 화면 개편은

- 바로 적용하는 작업이 아니라
- 먼저 로컬에서 보여주고
- 피드백으로 다듬고
- 괜찮으면 운영에 적용하는

**연구 기반 프로토타입 방식**으로 진행한다.
