# 요구사항 · 기획서 · 페이지 정체성 집중 정리

## 1. 이번 단계 목표

현재 전체 파이프라인은 아래처럼 길다.

1. 페이지 수집
2. 컴포넌트화
3. 디자인 MD 수집
4. 요구사항
5. 기획서
6. 디자인 빌더

하지만 현재는 이 전체를 한 번에 완성하려고 하면서, 각 단계의 품질과 연결성이 모두 흔들리고 있다.

이번 단계에서는 범위를 의도적으로 줄인다.

### 이번에 닫을 3개

1. **요구사항 입력**
2. **기획서(PRD)**
3. **페이지 정체성 프롬프트**

즉 이번 단계는 “빌더를 더 잘 돌리는 것”보다,

- 고객이 요구사항을 잘 넣을 수 있고
- 고객이 기획서를 잘 읽고 수정할 수 있고
- 페이지 정체성을 기준 문서처럼 유지할 수 있도록

UI와 문서 구조를 확실히 닫는 것이 목적이다.

---

## 2. 이번 단계에서 하지 않는 것

아래는 이번 단계의 직접 목표가 아니다.

- 디자인 빌더 결과 품질 고도화
- 디자인 MD를 builder rule object로 완전히 변환
- slot/component 시스템 전면 수정
- compare/version/pinned UX 정리
- AppFlowy 스타일 최종 마감

이 항목들은 다음 단계로 미룬다.

이번 단계에서는 위 요소들이 있더라도, **요구사항/기획서/정체성 흐름을 지원하는 최소 맥락**으로만 다룬다.

---

## 3. 현재 문제 정의

현재 코드 기준으로 문제는 아래 세 가지다.

### 3.1 요구사항 입력이 “기획서 입력”과 섞여 있음

현재 admin의 planner 영역은

- 고객 요청
- 레퍼런스
- tone
- avoid
- key message
- planning direction
- design direction

이 한 화면에 모두 섞여 있다.

그 결과 사용자는 “요구사항을 넣는 단계”와 “기획 문서를 편집하는 단계”를 구분하기 어렵다.

### 3.2 기획서가 고객 문서와 내부 AI 컨텍스트를 동시에 하려 함

현재 `requirementPlan`은 다음 둘을 동시에 하려 한다.

- 고객이 읽는 제안서
- builder가 읽는 구조화된 plan

이 때문에 문서는 길고 설명적이지만, 실무 단계에서 목적이 흐려진다.

### 3.3 페이지 정체성이 별도 기준 문서가 아니라 부가 설정처럼 보임

페이지 정체성은 이미 입력 UI가 있지만, 현재 흐름에서는

- planner 실행 전 필수 기준인지
- 기획서 어디에 반영되는지
- 고객이 왜 봐야 하는지

가 약하다.

즉 페이지 정체성이 “관리자 설정 모달”처럼 보이고, 기준 문서로 보이지 않는다.

---

## 4. 목표 구조

이번 단계에서는 화면과 문서를 아래 3개로 분리한다.

### 4.1 요구사항

역할:

- 고객/담당자가 이번 작업 요청을 정리하는 입력 단계

포함 항목:

- 요청 제목
- 요청 배경
- 핵심 메시지
- 원하는 방향
- 피해야 할 방향
- 참고 레퍼런스 URL
- 범위 선택
  - 페이지 전체
  - 특정 컴포넌트만

출력:

- `RequirementInput`
- 이후 기획서 초안의 재료

중요:

- 이 단계에서는 아직 긴 기획서를 직접 수정하지 않는다.
- “무엇을 원하느냐”만 분명히 입력한다.

### 4.2 기획서

역할:

- 고객/상위자/비개발 이해관계자가 읽고 합의하는 문서

포맷:

Claude 제안 기준으로 단일 PRD 스타일을 사용한다.

1. 한 줄 요약
2. 배경 / 문제 정의
3. 목표 범위
4. 핵심 기능
5. 사용자 흐름
6. 단계별 로드맵
7. 미결 사항 / 제외 판단 이유

출력:

- `RequirementDoc`

중요:

- 이 문서는 “잘 읽히는 문서”가 목적이다.
- builder용 세부 지시문은 여기서 분리한다.

### 4.3 페이지 정체성

역할:

- 이 페이지가 원래 무엇을 하는 페이지인지 고정하는 기준

포함 항목:

- 페이지 역할
- 페이지 목적
- 디자인 의도
- 반드시 유지할 것
- 피해야 할 것
- 시각 가드레일

출력:

- `PageIdentityBrief`

중요:

- planner 전에 항상 참조되는 기준이다.
- 기획서 상단의 “배경 / 문제 정의 / 범위”에 반영되어야 한다.

---

## 5. 기획서 포맷 적용 방식

Claude 제안 포맷을 현재 코드 구조에 맞추면 다음처럼 매핑한다.

### 5.1 현재 구조와의 매핑

현재 `requirementPlan`의 주요 필드:

- `title`
- `requestSummary`
- `planningDirection`
- `designDirection`
- `priority`
- `guardrails`
- `referenceNotes`
- `builderBrief`

이번 단계에서는 이를 아래처럼 정리한다.

| 현재 필드 | 새 문서 위치 | 비고 |
| --- | --- | --- |
| `title` | 한 줄 요약 | 유지 |
| `requestSummary` | 배경 / 문제 정의 | 재해석 |
| `planningDirection` | 목표 범위 / 핵심 방향 | 재구성 |
| `designDirection` | 핵심 기능 / 화면 방향 | 재구성 |
| `priority` | 단계별 로드맵 | 재해석 |
| `guardrails` | 제외 판단 / 유지 원칙 | 유지 |
| `referenceNotes` | 참고 자료 | 보조 부록 |
| `builderBrief` | 기획서에서 제거 | Build 단계로 이동 |

핵심은 이것이다.

- 기획서는 고객 문서다.
- `builderBrief`는 기획서가 아니라 빌더 단계 입력이다.

---

## 6. 화면 구조 원칙

이번 단계에서 UI는 아래처럼 간단하게 분리한다.

### 6.1 메뉴 구조

1. 페이지 정체성
2. 요구사항
3. 기획서

순서 이유:

- 페이지 정체성이 기준
- 요구사항은 이번 요청
- 기획서는 둘을 합쳐 만들어지는 산출물

### 6.2 각 화면 목적

#### 페이지 정체성 화면

- 기준 문서 보기/수정
- planner에 반영되는 원칙 확인
- “이 페이지는 원래 무엇인가”를 고정

#### 요구사항 화면

- 이번 요청 입력
- 레퍼런스 연결
- 범위 선택
- 기획서 생성 버튼

#### 기획서 화면

- 생성된 PRD 문서 보기
- 렌더 / 마크다운 전환
- 승인 / 반려
- 다음 단계 버튼
  - 디자인 계획으로 이동
  - 디자인 빌더 열기

---

## 7. 이번 단계 완료 기준

### 7.1 요구사항

- 사용자가 “이번 요청”만 입력하는 화면이 분리되어 있다.
- 기획서 편집 필드와 섞여 있지 않다.
- 레퍼런스 URL과 범위 선택이 요구사항 단계에 정리된다.

### 7.2 기획서

- 고객용 PRD 포맷으로 보인다.
- 문단 구조가 분명하다.
- 렌더와 원문 보기 둘 다 가능하다.
- `builderBrief` 같은 내부 빌더 지시문은 노출하지 않는다.

### 7.3 페이지 정체성

- 별도 메뉴/화면으로 보인다.
- planner 전에 반드시 참조되는 기준으로 표현된다.
- 기획서 상단에도 이 정체성이 자연스럽게 반영된다.

---

## 8. 바로 필요한 구현 순서

### 1단계

- 현재 planner 화면에서 `요구사항 입력`과 `기획서 편집`을 분리

### 2단계

- `RequirementDoc` 포맷을 PRD 구조로 재정렬

### 3단계

- `builderBrief`를 기획서 화면에서 제거하고 Build 단계 전용으로 이동

### 4단계

- 페이지 정체성 화면을 기준 문서처럼 보이게 승격

### 5단계

- 세 화면 간 이동 흐름 정리
  - 페이지 정체성 → 요구사항
  - 요구사항 → 기획서 생성
  - 기획서 → 디자인 계획 / 디자인 빌더

---

## 9. 결론

이번 단계의 핵심은 전체 파이프라인을 다 고치는 것이 아니다.

오히려 반대로,

- 요구사항
- 기획서
- 페이지 정체성

이 세 가지를 **고객이 실제로 읽고 쓰고 합의할 수 있는 수준으로 먼저 닫는 것**이다.

즉 지금은 “builder를 더 똑똑하게 만들기”보다,

**좋은 입력과 좋은 기준 문서를 먼저 만드는 단계**로 보는 것이 맞다.
