# Admin Preview Workbench Structure

이 문서는 `admin` 화면을 어떤 개념과 흐름으로 운영할지 정리한 기준안이다.

핵심은 `홈페이지 유지보수`가 아니라, `고객 요청을 빠르게 시안화해서 눈으로 검토하는 프리뷰 워크벤치`라는 점이다.

---

## 1. 제품 성격

`admin`의 1차 목적은 아래 3가지다.

1. 고객 요구사항을 정리한다.
2. 그 요구사항에 맞는 기획/디자인 방향을 빠르게 시안화한다.
3. 생성된 버전을 저장하고, 클론 페이지에 보여줄 대표 버전을 고정한다.

즉 이 화면은 CMS나 정교한 운영툴이 아니라, 아래 흐름을 위한 워크벤치다.

1. 대상 페이지 선택
2. 요구사항 정리
3. 시안 생성
4. 버전 저장
5. View 고정
6. 작업 이력 호출 및 비교

이때 AI 역할은 하나로 뭉치지 않고 아래 2단계로 나눈다.

1. `Planner LLM`
2. `Builder LLM`

---

## 2. 화면의 큰 구조

전체 화면은 `좌측 탐색 영역 + 우측 작업 영역`으로 본다.

### 2.1 좌측 탐색 영역

좌측은 계속 고정되는 탐색 영역이다.

역할:

1. 어떤 페이지를 작업할지 고른다.
2. 현재 페이지 상태를 빠르게 본다.
3. 저장된 버전 이력을 확인하고 다시 불러온다.

좌측 섹션 순서:

1. `수정할 페이지 선택`
2. `페이지 상태`
3. `시안 버전 이력`

### 2.2 우측 작업 영역

우측은 실제 작업이 일어나는 메인 영역이다.

역할:

1. 현재 작업 컨텍스트를 보여준다.
2. 요구사항을 정리한다.
3. AI로 시안을 만든다.
4. 시안을 저장하고 View에 고정한다.
5. 필요할 때만 기술/컴포넌트 정보를 펼쳐본다.

우측 섹션 순서:

1. `기획/시안 작업` 헤더
2. `요구사항 정리`
3. `시안 생성`
4. `생성 결과 / 버전 저장`
5. `보조 정보`

---

## 3. 사용자 흐름

화면을 보는 흐름은 아래 순서를 유지한다.

1. 좌측 상단에서 페이지를 고른다.
2. `선택하기`를 눌러 해당 페이지를 작업 대상으로 불러온다.
3. 우측 상단에서 현재 어떤 페이지/버전을 보고 있는지 확인한다.
4. `요구사항 정리`에서 고객 요청과 레퍼런스를 입력한다.
5. AI가 정리한 요구사항/기획 방향을 사용자가 수정한다.
6. 그 정리본을 기준으로 `시안 생성`을 실행한다.
7. 생성된 결과를 새 버전으로 저장한다.
8. 저장된 버전 중 하나를 `현재 View로 고정`한다.
9. 좌측 `시안 버전 이력`에서 이전 버전을 다시 불러오거나 비교한다.

이 순서가 화면 배치와 다르면 안 된다.

AI 실행 흐름도 같은 순서를 따른다.

1. `Planner LLM`이 요구사항을 정리한다.
2. 사용자가 정리 결과를 검토하고 수정한다.
3. `Builder LLM`이 확정된 정리본을 기준으로 시안을 생성한다.
4. 생성 결과를 버전으로 저장하고 필요 시 View에 고정한다.

## 5. 우측 섹션 상세

### 5.1 기획/시안 작업 헤더

화면 위치:

- 우측 메인 영역 맨 위

설명:

- 현재 어떤 페이지와 어떤 버전을 대상으로 작업 중인지 보여주는 헤더
- 우측 상단에는 현재 페이지 관련 액션을 둔다

들어갈 항목:

1. 페이지명
2. 현재 작업 버전
3. 현재 View 고정 버전
4. 우측 상단 액션 버튼

권장 문구:

- 헤더명: `기획/시안 작업`
- 정보 라벨:
  - `현재 작업 버전`
  - `현재 View 고정 버전`
- 버튼:
  - `기준 화면 보기`
  - `현재 버전 보기`
  - `링크 복사`
  - `새 버전 저장`

### 5.2 요구사항 정리

화면 위치:

- 헤더 바로 아래
- 우측 본문에서 가장 먼저 크게 보이는 핵심 섹션

설명:

- 이 화면의 중심 기능
- 시안을 만들기 전에 무엇을 만들지 먼저 정리하는 영역

입력 방식:

1. `직접 작성`
2. `레퍼런스 사이트 입력`
3. `직접 작성 + 레퍼런스 사이트`

입력 항목:

1. 고객 요청
2. 중요 메시지
3. 원하는 방향
4. 피해야 할 방향
5. 참고 사이트 URL
6. 톤앤매너

AI 정리 결과:

1. 요구사항 요약
2. 기획 방향
3. 디자인 방향
4. 적용 우선순위
5. 주의사항

원칙:

1. AI가 정리한 결과는 사용자가 직접 수정할 수 있어야 한다.
2. 정리 결과는 시안 생성의 기준 데이터가 된다.
3. 이 단계의 기본 담당 모델은 `Planner LLM`이다.

권장 문구:

- 섹션명: `요구사항 정리`
- 입력 방식 라벨: `정리 방식`
- 실행 버튼: `AI로 요구사항 정리`
- 결과 영역 제목:
  - `요구사항 요약`
  - `기획 방향`
  - `디자인 방향`
  - `적용 우선순위`
  - `주의사항`
- 보조 문구: `AI가 정리한 내용을 검토하고 직접 수정할 수 있습니다.`

### 5.4 생성 결과 / 버전 저장

화면 위치:

설명:

1. 생성 직후 결과는 임시 작업 결과다.
2. 저장은 항상 `새 버전 저장` 기준으로 간다.
3. 저장된 버전 중 하나만 `현재 View`로 고정할 수 있다.
4. 클론 페이지는 항상 `현재 View`로 고정된 버전만 노출한다.

들어갈 항목:

1. 생성 결과 미리보기

3. 저장 버튼
4. View 고정 버튼
5. 버전 비교 버튼

권장 문구:

- 섹션명: `생성 결과 / 버전 저장`
- 버튼:
  - `새 버전 저장`
  - `현재 View로 고정`
  - `이전 버전과 비교`
- 상태 문구:
  - `아직 저장되지 않은 작업 결과입니다.`
  - `현재 View로 고정된 버전입니다.`

## 6. 버전 개념 정리

버전은 아래 2종류로 구분한다.

### 6.1 작업 버전

설명:

- AI가 생성하고 저장한 개별 시안 버전
- 여러 개 저장 가능
- 이력에서 비교/호출 가능

예시:

- `home_v1`
- `home_campaign_a`
- `home_req_20260413_01`

### 6.2 View 고정 버전

설명:

- 클론 페이지에서 실제로 노출되는 대표 버전
- 저장된 작업 버전 중 하나를 선택해서 고정한다

원칙:

1. 작업 버전과 View 고정 버전은 다를 수 있다.
2. 사용자가 새 시안을 만들어도, View 고정을 바꾸기 전까지 클론 페이지는 기존 버전을 유지한다.
3. 클론 페이지는 임시 생성본이 아니라 `현재 View로 고정`된 버전만 보여준다.

권장 상태명:

1. `현재 작업 버전`
2. `현재 View 고정 버전`
3. `현재 View`

## 7. LLM 역할 분리

시안 워크벤치의 AI는 하나의 범용 역할로 다루지 않고, 아래 2개의 책임으로 나눈다.

### 7.1 Planner LLM

역할:

1. 고객 입력과 레퍼런스를 해석한다.
2. 모호한 요구사항을 구조화한다.
3. 기획 방향과 디자인 방향을 정리한다.
4. 사람이 수정 가능한 초안 문서를 만든다.

입력:

1. 사용자 직접 작성 내용
2. 참고 URL
3. 현재 페이지 정보
4. 기존 콘텐츠/구조 정보

출력:

1. 요구사항 요약
2. 기획 방향
3. 디자인 방향
4. 적용 우선순위
5. 주의사항

원칙:

1. Planner 결과는 반드시 사용자가 수정 가능해야 한다.
2. Planner는 patch를 직접 확정하는 주체가 아니다.

### 7.2 Builder LLM

역할:

1. 확정된 요구사항 정리본을 실제 페이지 구조에 매핑한다.
2. 허용된 컴포넌트/슬롯 범위 안에서 시안을 만든다.
3. 도구 호출과 patch 생성을 통해 저장 가능한 버전을 만든다.

입력:

1. 사용자가 확정한 요구사항 정리본
2. 페이지 slot registry
3. editable component inventory
4. design rules / tool context

출력:

1. 시안 결과
2. 저장 가능한 버전 데이터
3. 생성 근거 요약

원칙:

1. Builder는 원문 고객 요청을 다시 해석하는 주체가 아니라, 확정된 정리본을 실행하는 주체다.
2. Builder는 workspace 규칙과 저장 구조를 반드시 따라야 한다.

### 7.3 작업 리스트 반영

향후 구현 작업에는 아래 항목을 포함한다.

1. `Planner LLM`과 `Builder LLM`의 시스템 역할 분리
2. Planner 출력 스키마 정의
3. Builder 입력/출력 스키마 정의
4. 사용자가 Planner 결과를 수정/승인한 뒤 Builder를 호출하는 흐름 정의
5. 작업 이력에 Planner 정리본과 Builder 생성 결과를 함께 연결하는 구조 정의

## 8. 문구 치환 기준

기존 편집 중심 문구보다 아래 문구를 우선 사용한다.

1. `페이지 상세` -> `기획/시안 작업`
2. `작업 이력` -> `시안 버전 이력`
3. `수정 가능한 컴포넌트` -> `보조 정보 > 수정 가능한 컴포넌트`
4. `AI로 적용` -> 상황에 따라 아래로 분리
   - `AI로 요구사항 정리`
   - `AI로 시안 만들기`

## 9. 현재 기준 결론

이 화면의 중심은 `컴포넌트 수정`이 아니라 아래 순서다.

1. 요구사항 해석
2. 기획/디자인 방향 정리
