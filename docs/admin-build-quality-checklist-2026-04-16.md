# 관리자 화면 빌드 품질 점검 체크리스트

작성일: 2026-04-16

## 1. 목적

이 문서는 운영자가 `admin` 화면에서 현재 페이지의 빌드 준비 상태를 점검할 때 실제로 확인해야 할 항목을 정리한 체크리스트다.

기준:
- `page identity`
- `requirements`
- `PRD`
- `builder`
- `final acceptance`

이 5개를 순서대로 본다.

---

## 2. 페이지 정체성

확인 항목:

- 페이지 역할이 한 줄로 설명된다
- 페이지 목적이 비어 있지 않다
- 디자인 의도가 비어 있지 않다
- `반드시 유지할 것`과 `피해야 할 것`이 각각 최소 1개 이상 있다
- 시각 가드레일이 존재한다
- 최근 수정 시각 또는 최근 작업 이력이 보인다

실패로 보는 경우:

- 편집 값은 있는데 읽기 화면에 반영되지 않는다
- 필드는 채워졌지만 이 페이지가 원래 무엇을 하는지 판단이 안 된다
- PRD와 builder 판단에 쓸 수 없는 추상 문장만 있다

운영 의미:

- 이 단계가 비어 있으면 이후 문서와 빌드 결과 품질 판정도 불안정하다

---

## 3. 요구사항

확인 항목:

- 요청 제목이 있다
- 요청 배경 또는 변경 이유가 있다
- 핵심 메시지가 있다
- 원하는 방향이 있다
- 피해야 할 방향이 있다
- 레퍼런스 URL이 있거나, 없는 이유가 분명하다
- 변화 강도(`low/medium/high`)가 설정돼 있다
- 저장 후 다시 진입했을 때 값이 유지된다
- “요구사항 저장”과 “기획서 생성”의 역할이 UI에서 구분된다

실패로 보는 경우:

- 요구사항 입력이 곧 PRD 편집처럼 보인다
- 저장 후 값이 일부 사라진다
- 사용자가 지금 무엇을 입력하는 단계인지 설명이 안 된다

운영 의미:

- 이 단계는 builder 실행 전 입력 계약이다

---

## 4. PRD

확인 항목:

- PRD가 비어 있을 때 빈 상태 메시지가 다음 행동을 안내한다
- PRD가 있을 때 문서처럼 읽힌다
- 최소 아래 항목이 구분된다
  - 한 줄 요약
  - 배경 / 문제 정의
  - 목표 범위
  - 핵심 방향 또는 기능
  - 우선순위 / 로드맵
  - 가드레일 / 제외 항목
- PRD가 요구사항과 페이지 정체성을 반영한다
- `마크다운 복사`, `편집`, `빌더 실행` 동선이 자연스럽다
- 운영 데이터 기준 필드가 누락되지 않는다

실패로 보는 경우:

- raw JSON처럼 보인다
- 읽는 문서와 builder용 내부 데이터가 뒤섞여 이해가 어렵다
- 요구사항은 있는데 PRD가 왜 이렇게 나왔는지 연결이 안 된다
- `builderBrief`, `priority`, `guardrails` 등 운영상 중요한 필드가 빠진다

운영 의미:

- 이 단계는 “빌드를 돌려도 되는 승인 문서가 있는가”를 보는 단계다

---

## 5. Builder 실행 준비

확인 항목:

- builder 실행 버튼이 현재 상태에 맞는 문구를 보여준다
  - 요구사항 작성 필요
  - 기획서 생성 필요
  - 실행 가능
- 최신 draft build가 있는지 보인다
- 저장 버전이 있는지 보인다
- preview / compare 진입점이 있다
- 현재 빌드가 어느 페이지/뷰포트 기준인지 혼동이 없다
- `designSpecMarkdown` 또는 동급 실행 스펙이 준비되어 있다
- 섹션 단위 실행 기준이 있음을 설명할 수 있다

실패로 보는 경우:

- PRD는 있는데 builder를 왜 못 돌리는지 이유가 안 보인다
- builder 결과가 있는데 어떤 입력 기준으로 생성됐는지 추적이 안 된다
- preview/compare 링크가 없거나 현재 선택과 맞지 않는다

운영 의미:

- 이 단계는 “문서가 아니라 실제 실행 준비가 끝났는가”를 보는 단계다

---

## 6. 결과물 품질

확인 항목:

- 최신 draft 또는 version이 존재한다
- 최근 생성 시각과 요약이 보인다
- 실제 화면 preview가 열린다
- compare 화면으로 이동 가능하다
- 필요 시 pinned/current 상태를 구분할 수 있다
- 빌더 실패나 timeout이 있으면 운영자가 바로 알 수 있다

실패로 보는 경우:

- 결과물은 저장돼 있는데 어떤 것이 최신인지 알 수 없다
- compare 없이 결과만 보여서 품질 판정이 어렵다
- 에러는 로그에만 있고 운영 화면에서는 보이지 않는다

운영 의미:

- 이 단계는 “builder가 실행되었다”가 아니라 “결과가 검수 가능한 형태인가”를 본다

---

## 7. Final Acceptance

확인 항목:

- 현재 페이지의 acceptance 상태가 보인다
- 다음 pending bundle 또는 다음 actionable 대상이 보인다
- compare 링크로 바로 이동할 수 있다
- `pass/fail/pending` 저장 흐름이 존재한다
- `fail`이면 note를 남겨야 한다
- `ready-for-llm`인지, 아니면 무엇 때문에 막혀 있는지 설명된다

실패로 보는 경우:

- builder 이후 어디로 가야 하는지 admin에서 끊긴다
- acceptance 상태가 detail 깊은 곳에만 있어서 첫 진입에서 판단이 안 된다
- `blocked-by-acceptance`와 `ready-for-llm` 구분이 안 된다

운영 의미:

- 이 단계가 있어야 `admin`이 단순 작업 화면이 아니라 최종 빌드 진입 게이트가 된다

---

## 8. 상단 요약 바에 반드시 보여야 할 정보

권장 고정 항목:

1. `페이지 정체성`
2. `요구사항`
3. `PRD`
4. `Builder`
5. `Acceptance`

각 항목은 최소 아래 중 하나의 상태로 보여야 한다.

- 없음
- 작성 중
- 준비됨
- 결과 있음
- 검수 필요
- 완료

중요:
- 현재 `identity / requirements / prd / builder`까지만 보이면 운영 흐름이 builder에서 끊긴다
- `Acceptance`까지 들어가야 최종 빌드 준비도 점검이라는 목적과 맞다

---

## 9. 페이지 리스트에서 바로 보여야 할 정보

목록 레벨 최소 항목:

1. 페이지명
2. 최근 작업 위치
3. 정체성 / 요구사항 / PRD 상태 점
4. builder 결과 존재 여부
5. acceptance 이슈 또는 다음 작업 필요 여부

이유:

- 운영자는 상세 페이지에 들어가기 전에도 어디가 막혀 있는지 우선순위를 잡아야 한다

---

## 10. 이번 범위에서 가장 중요한 결론

`admin` 품질 점검의 핵심은 “버튼이 많다/적다”가 아니다.

핵심은 아래 질문에 즉시 답할 수 있느냐다.

1. 이 페이지는 지금 빌드를 돌릴 준비가 되었는가
2. 아직 안 되었다면 어디가 비어 있는가
3. builder는 실제로 실행 가능한가
4. 결과물은 검수 가능한가
5. 최종 acceptance로 넘겨도 되는가

이 5개가 한 화면 흐름에서 판정되면 이번 `admin` 개편 목적에 부합한다.
