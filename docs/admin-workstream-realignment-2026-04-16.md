# 관리자 작업축 재정렬 문서

작성일: 2026-04-16

## 1. 목적

이 문서는 현재 `admin` 관련 변경이 여러 층위의 작업으로 섞여 있는 상태를 다시 분리하기 위해 작성한다.

지금 워크트리에는 아래 성격이 한 번에 섞여 있다.

1. `admin` UX 재구성
2. Planner / requirement plan 저장 포맷 정리
3. builder-ready / readiness 운영 검증 체계
4. `home marketing-area` 실제 blocker 복구
5. LLM timeout / fallback 안정화

문제는 이 5가지가 서로 관련은 있지만, 같은 우선순위와 같은 승인 기준으로 다뤄지면 안 된다는 점이다.

이 문서의 목적은:

1. 각 작업축의 목적을 다시 고정하고
2. 무엇이 사용자 요구사항과 직접 연결되는지 구분하고
3. 무엇이 운영 확장인지 분리하고
4. 이후 커밋 / 문서 / 검토 단위를 다시 자를 수 있게 만드는 것이다.

---

## 2. 이번 재정렬의 기본 원칙

### 원칙 A. 문서 중심 요구사항과 운영 확장을 분리한다

현재 사용자의 최종 요구사항 축은 우선 아래 3개였다.

1. 페이지 정체성
2. 요구사항
3. 기획서

즉 이번 단계의 전면 요구는 `admin`을 이 세 축 중심으로 다시 읽히게 만드는 것이었다.

따라서 아래 항목은 직접 요구사항 범위다.

- UX 구조
- 문서 흐름
- Planner 저장 결과의 정리

반면 아래 항목은 운영 확장 성격이 강하다.

- full completion score
- readiness audit
- sidecar audit
- builder governance
- 전체 15페이지 상태판

이 둘은 같이 존재할 수는 있지만, 같은 커밋과 같은 승인 기준으로 섞이면 다시 꼬인다.

### 원칙 B. 실제 blocker fix와 구조 개편을 분리한다

`marketing-area` 복구처럼 실제 동작 blocker를 닫는 일은 UX 재구성과 별개로 판정해야 한다.

### 원칙 C. 안정화 패치는 별도 판단 단위로 둔다

timeout / fallback 조정은 운영 안정성에는 중요하지만, 요구사항 문서 흐름을 바꾸는 작업은 아니다.

## 3. 작업축 1 — Admin UX 재구성

### 3.1 이 작업의 본래 목적

이 작업축의 목적은 `admin`을 단순 운영 콘솔처럼 보이게 하는 것이 아니라,

1. 페이지를 선택하고
2. 페이지 정체성을 확인하고
3. 요구사항을 입력하고
4. 기획서를 읽고
5. 그 다음 단계로 Builder를 연결하는

`preview workbench`로 다시 보이게 만드는 것이다.

즉 여기서의 핵심은 기능 추가가 아니라 **위계 재정렬**이다.

### 3.2 여기까지 포함되는 범위

- 좌측 페이지 선택 구조
- 상단 상태 요약 바
- 본문 3축:
  - 페이지 정체성
  - 요구사항
  - 기획서
- 후순위 영역:
  - 관련 화면
  - 버전 확인
  - 보조 기능
  - 운영 도구

### 3.3 현재 왜 꼬였는가

현재 구현은 문서 3축 재정렬은 맞게 들어갔지만, 동시에 운영 상태판과 빌더 메타가 다시 전면으로 올라왔다.

즉 방향은 맞았지만, 아래 두 성격이 다시 섞였다.

1. `문서 중심 화면 재구성`
2. `운영 통제 화면 확장`

### 3.4 이 작업축에서 맞춰야 할 방향

이 작업축에서 확정해야 할 기준은 아래다.

1. 이번 단계의 메인 화면에서 가장 먼저 보여야 하는 것은 `정체성 / 요구사항 / 기획서`인가
2. 디자인 빌더는 기본 노출인지, 기획서 승인 이후 조건부 노출인지
3. related / version / support / ops는 항상 보이는지, 일부는 접힘인지
4. 상태 요약 바는 문서 상태만 보여주는지, full completion 같은 운영 메타까지 함께 보여주는지

### 3.5 현재 권장 방향

현재 단계 기준 권장 방향은 아래다.

1. 메인 전면은 `정체성 / 요구사항 / 기획서`
2. 디자인 빌더는 `기획서 승인 + 실행` 이후에만 본문 하단에서 열린다
3. 관련 화면 / 버전 확인은 aside
4. 보조 기능 / 운영 도구는 접힘 유지
5. full completion 계열은 메인 문서 흐름이 아니라 별도 운영 카드 또는 별도 화면으로 분리
6. 디자인 빌더가 완료되면 전체 페이지 iframe을 전면에 다시 보여주는 것이 아니라, `실제로 변경된 섹션 디자인 미리보기`만 아래에 노출한다

### 3.5-a 디자인 빌더 UX 확정안

사용자 정의 기준으로, 디자인 빌더 UX는 아래 흐름으로 고정한다.

1. 사용자는 먼저 `정체성 -> 요구사항 -> 기획서` 흐름을 완료한다.
2. 기획서가 승인되기 전에는 디자인 빌더를 전면 작업 영역으로 띄우지 않는다.
3. 기획서가 승인되고 사용자가 실행을 시작하면, 디자인 빌더 섹션이 `기획서 아래`에 나타난다.
4. 빌더 실행 중에는 진행 상태와 현재 처리 단계가 보인다.
5. 빌더 완료 후에는 `변경된 섹션만` 디자인 미리보기 카드 또는 섹션 프리뷰로 노출한다.
6. 변경되지 않은 섹션은 여기서 반복 노출하지 않는다.

즉 이 UX에서 디자인 빌더는:

- 독립 메인 화면이 아니라
- 기획서 다음 단계로 이어지는 실행 레이어이며
- 결과도 전체 페이지 재노출이 아니라 `변경분 중심 결과 확인`에 초점을 둔다.

### 3.6 문서 확정용 한 줄

`Admin UX 재구성의 목표는 문서 3축(정체성/요구사항/기획서)을 전면에 두고, 디자인 빌더는 기획서 승인 후 하단 실행 레이어로 열리며 완료 후에는 변경된 섹션 미리보기만 노출하는 것이다.`

### 3.7 후속 요구사항 입력 스키마

현재 UX에서 빠진 핵심은 `개입 범위 선택`과 `패치 깊이 선택`이다.

후속 기준 문서는 아래다.

- [관리자 요구사항 범위/패치 깊이 스키마](./admin-requirement-scope-schema-2026-04-17.md)
- [관리자 전면개편 단계형 완료 기준](./admin-full-redesign-delivery-gates-2026-04-17.md)

이 문서 기준으로 이후 요구사항 입력은:

1. 페이지 선택
2. 개입 계층 선택
3. 패치 깊이 선택
4. 요구사항 입력
5. 기획서 생성
6. 빌더 실행

흐름으로 정리한다.

## 4. 작업축 2 — Planner / Requirement Plan 저장 포맷 정리

### 4.1 이 작업의 본래 목적

이 작업축의 목적은 사용자 요구사항과 Planner 결과를 나중에도 안정적으로 읽을 수 있게 저장 구조를 고정하는 것이다.

즉:

1. 중복 줄 제거
2. 길이 제한
3. priority / guardrails / builderBrief / sectionBlueprints 구조 정리
4. 저장 시 최소한의 normalize

를 통해 history 품질을 높이는 작업이다.

### 4.2 여기까지 포함되는 범위

- `sanitizePlanLineArray`
- `sanitizeRequirementPlanOutput`
- `normalizeRequirementPlan`
- requirement plan 저장 시 output 정리

### 4.3 현재 왜 꼬였는가

이 작업은 원래 `history 품질 보정`인데,
현재는 `사용자 최종 요구사항 원문`과 `backfill로 생성한 실행 문서`가 같은 requirement history처럼 보이게 만들 위험이 있다.

즉 저장 포맷 정리는 맞지만,
무엇이

- 사용자가 직접 승인한 요구사항인지
- 자동 문서화 결과인지

가 분리되어야 한다.

### 4.4 이 작업축에서 맞춰야 할 방향

이 축에서 확정해야 할 것은 아래다.

1. requirement history는 `사용자 입력 원문`과 `planner output`을 명확히 구분할 것인가
2. 자동 backfill 산출물은 requirement plan으로 저장해도 되는가
3. 자동 생성 산출물에는 `generatedBy`, `sourceType`, `approvalState` 같은 구분 필드가 더 필요한가
4. sanitize는 허용하되, 원문 보존 레이어는 따로 둘 것인가

### 4.5 현재 권장 방향

현재 기준 권장 방향은 아래다.

1. requirement plan 저장 포맷 정리는 유지
2. 다만 `user-authored`와 `auto-backfilled`를 구분하는 메타를 분리
3. UI에서도 `사용자 확정 요구사항`과 `자동 정리 문서`를 같은 위계로 보이지 않게 함

### 4.6 문서 확정용 한 줄

`Requirement plan 저장 포맷은 정리하되, 사용자 원문 요구사항과 자동 문서화 산출물은 반드시 구분한다.`

## 5. 작업축 3 — Builder-ready / Readiness 운영 검증 체계

### 5.1 이 작업의 본래 목적

이 작업축의 목적은 15개 페이지 전체에 대해:

1. 현재 clone이 열리는가
2. sidecar가 충분한가
3. builder issue가 없는가
4. docs가 latest 포맷인가
5. full completion 기준을 통과하는가

를 운영 기준으로 실측하고 수치화하는 것이다.

이건 UX 작업이 아니라 **운영 검증 시스템**이다.

중요하게, 이 작업축은 **고객용 문서 흐름의 일부가 아니다.**

즉 아래에는 포함되지 않는다.

1. 고객이 읽는 기획서
2. 고객 승인 전 문서 흐름
3. 디자인 빌더 결과 확인 UX

이 작업축은 내부 운영자가:

- 현재 빌드 가능 상태인지
- 어떤 blocker가 있는지
- 어떤 페이지가 준비되지 않았는지

를 판정하기 위한 내부 도구다.

### 5.2 여기까지 포함되는 범위

- `readiness-audit`
- `artifact-sidecar-audit`
- `fullCompletionReport`
- status board / runbook
- backfill docs script

### 5.3 현재 왜 꼬였는가

이 축은 운영상 매우 유용하지만,
이번 `admin UX` 요구사항에 바로 필요한 전면 과업은 아니었다.

즉 이 작업은 틀린 것이 아니라, **들어온 타이밍과 전면 노출 위치가 과도했다**.

### 5.4 이 작업축에서 맞춰야 할 방향

이 축에서 확정해야 할 것은 아래다.

1. full completion은 이번 단계의 필수 승인 기준인가, 별도 운영 지표인가
2. `admin` 메인 화면에서 항상 보여줄 것인가, 운영 전용 영역에서 볼 것인가
3. 15페이지 전체 상태판은 UX 개편과 같은 커밋으로 묶을 것인가
4. docs backfill은 사용자 요구사항 정리와 별도 운영 스크립트로 분리할 것인가

### 5.5 현재 권장 방향

현재 기준 권장 방향은 아래다.

1. readiness / full completion은 별도 운영 레이어로 분리
2. 고객용 `admin` 메인 흐름에서는 기본적으로 노출하지 않는다
3. 상태판 문서와 런북은 UX 최종안과 분리 커밋
4. backfill docs는 운영용 maintenance script로 분리
5. 필요하면 `운영 전용 도구` 접힘 영역 또는 별도 운영 화면에서만 확인한다
6. 고객에게 보여주는 기획서/빌더 UX에서는 `판정 문서`, `점수표`, `full completion 통과 여부`를 직접 노출하지 않는다

### 5.6 문서 확정용 한 줄

`Builder-ready 및 readiness 체계는 유지하되, 고객 UX 범위에서는 제외하고 별도 운영 검증 레이어로만 다룬다.`

## 6. 작업축 4 — Home marketing-area blocker 복구

### 6.1 이 작업의 본래 목적

이 작업축의 목적은 `home` 계열 full completion을 막고 있던 실제 구조 blocker를 닫는 것이다.

구체적으로는:

- `marketing-area` slot registry 누락 복구
- fallback registry 병합
- home builder blocker 제거

가 핵심이다.

### 6.2 여기까지 포함되는 범위

- default slot registry 보강
- `marketing-area` source 등록
- registry fallback merge

### 6.3 현재 왜 꼬였는가

이건 실제 버그/누락 수정인데,
현재 문서화/운영 검증/UX 개편 흐름 안에 같이 묶여 있어서
마치 큰 구조 개편의 일부처럼 보인다.

하지만 성격상 이건 가장 독립적인 blocker fix다.

### 6.4 이 작업축에서 맞춰야 할 방향

이 축에서는 아래만 확인하면 된다.

1. `marketing-area` 복구는 지금 바로 독립 fix로 인정할 것인가
2. 이 fix가 full completion 문서와 별도여도 되는가
3. `home` 전용 hotfix로 먼저 정리할 것인가

### 6.5 현재 권장 방향

현재 기준 권장 방향은 아래다.

1. `marketing-area` 복구는 독립 bugfix로 분리
2. readiness 문서 서술과는 분리
3. 결과적으로 full completion에 영향을 주더라도, 변경 목적은 `운영 문서화`가 아니라 `실제 blocker 제거`로 기록

### 6.6 문서 확정용 한 줄

`Home marketing-area 복구는 문서 개편과 분리된 독립 blocker fix로 다룬다.`

## 7. 작업축 5 — LLM timeout / fallback 안정화

### 7.1 이 작업의 본래 목적

이 작업축의 목적은 Planner / Builder LLM 호출이 너무 쉽게 timeout 나거나 단일 모델 실패로 끊기는 문제를 줄이는 것이다.

즉:

- timeout 상향
- fallback model 시도
- narrative 길이 상한 조정

같은 안정화가 여기에 해당한다.

### 7.2 여기까지 포함되는 범위

- `OPENROUTER_TIMEOUT_MS`
- `LLM_REQUEST_TIMEOUT_MS`
- builder fallback model
- narrative bounding

### 7.3 현재 왜 꼬였는가

이건 운영 안정성 패치인데,
현재는 requirement plan 정리와 문서 흐름 변경과 같이 보이고 있다.

그러면 검토할 때 아래가 동시에 섞인다.

1. 기능 방향이 맞는가
2. UI 흐름이 맞는가
3. 모델 안정성이 좋아졌는가

이 셋은 같은 질문이 아니다.

### 7.4 이 작업축에서 맞춰야 할 방향

이 축에서 확정해야 할 것은 아래다.

1. timeout 상향은 현재 운영에서 필요한가
2. fallback model 사용이 정책상 허용되는가
3. 이 안정화 패치를 UX 정리와 함께 묶을 필요가 있는가

### 7.5 현재 권장 방향

현재 기준 권장 방향은 아래다.

## 9. 이번 문서의 임시 확정안

현재까지의 권장 임시 확정안은 아래다.

1. `Admin UX 재구성`은 이번 요구사항의 본체로 본다.
2. `Requirement plan 저장 포맷 정리`는 이번 요구사항 지원축으로 본다.
3. `Home marketing-area 복구`는 독립 blocker fix로 분리한다.
4. `LLM timeout / fallback 안정화`는 독립 안정화 패치로 분리한다.
5. `Builder-ready / readiness 운영 검증 체계`는 고객 UX 범위에서 제외하고 별도 운영 확장으로 분리한다.
6. 디자인 빌더 UX는 `기획서 승인 후 실행 -> 하단 노출 -> 변경된 섹션 미리보기만 표시`로 본다.

### 9.1 고객용 / 내부용 최종 분리

현재 기준 확정안은 아래와 같다.

#### 고객용 전면 흐름

1. 페이지 정체성
2. 요구사항
3. 기획서
4. 기획서 승인 후 디자인 빌더 실행
5. 변경된 섹션 디자인 미리보기 확인

1. readiness audit
2. full completion report
3. artifact sidecar audit
4. builder blocker 판정
5. 전체 페이지 상태판 / 런북

즉 고객용 흐름에는 `판정 문서`가 직접 들어가지 않는다.
