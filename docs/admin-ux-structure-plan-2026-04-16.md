# 관리자 화면 UX 구조 계획

## 1. 목적

현재 관리자 화면은 기능은 많지만, 사용자 입장에서

- 어디서 무엇을 시작해야 하는지
- 어떤 정보가 기준 문서인지
- 어떤 화면이 이번 단계의 핵심인지

가 명확하지 않다.

특히 이번 단계의 목표는 전체 파이프라인을 다 다듬는 것이 아니라,

1. 페이지 정체성
2. 요구사항
3. 기획서

이 세 가지를 고객과 내부 담당자가 잘 읽고 잘 입력할 수 있도록 만드는 것이다.

따라서 관리자 화면도 이 세 가지를 중심으로 다시 구조화해야 한다.

---

## 2. 현재 문제

현재 `web/admin.html`은 기본적으로 아래 구조다.

- 좌측: 페이지 목록 + 작업 이력
- 우측: 페이지 상세

이 큰 틀은 괜찮다.

문제는 우측 상세 안에 아래가 한꺼번에 섞여 있다는 점이다.

- 페이지 정체성
- planner 입력
- planner 초안 편집
- 고객용 기획서 요약
- builder 실행
- 비교 링크
- 버전 관리
- 각종 runtime / acceptance / coverage / artifact 메타

즉 “이번에 봐야 할 것”과 “운영자가 나중에 보는 것”이 같은 위계로 쌓여 있다.

그 결과:

- 화면이 길고
- 섹션 우선순위가 모호하며
- planner와 builder가 한 덩어리로 보이고
- 페이지 정체성이 기준 문서처럼 느껴지지 않는다

---

## 3. 외부 기준에서 얻을 수 있는 방향

### 3.1 좌측 내비는 목적지 선택용

좌측 사이드바는 현재 페이지 목록 선택에 집중하는 것이 맞다.

좋은 관리자 화면은 좌측에서

- “어느 페이지를 보고 있는가”
- “어느 페이지가 작업 중인가”

만 빠르게 파악할 수 있어야 한다.

즉 좌측은 **페이지 선택용**으로 유지하고, 상세 내부 내비는 우측에서 처리한다.

### 3.2 긴 상세 화면에는 내부 목차가 필요하다

우측 상세가 길다면, 사용자는 현재 위치를 잃기 쉽다.

따라서 상세 상단에는

- 현재 페이지의 주요 섹션으로 이동하는
- 고정형 앵커 탭

이 필요하다.

이번 단계에서는 아래 세 섹션만 앵커 대상이 된다.

- 페이지 정체성
- 요구사항
- 기획서

### 3.3 빌더/비교/운영 메타는 이번 단계의 전면 영역이 아니다

현재 화면에서는 빌더와 버전 영역이 planner와 거의 같은 비중으로 보인다.

하지만 이번 단계 목표는 빌더가 아니라:

- 고객이 잘 입력하는 요구사항
- 고객이 읽는 기획서
- 페이지 정체성 기준

이다.

따라서 builder/compare/acceptance/runtime/debug는 메인 영역에서 비중을 낮춰야 한다.

## 4. 목표 구조

## 4.1 전체 프레임

### 좌측

- 페이지 목록
- 검색
- 간단한 상태 필터
- 최근 작업/기록은 보조영역

### 우측

- 페이지 헤더
- sticky anchor
- 본문 3개 섹션
  1. 페이지 정체성
  2. 요구사항
  3. 기획서

### 후순위 영역

- 디자인 빌더
- 비교
- 버전 관리
- runtime / acceptance / coverage / artifact

이 영역은 기본 접힘 또는 별도 하단 운영 섹션으로 이동

---

## 5. 좌측 목록 구조

좌측은 “페이지 선택” 역할에만 집중한다.

### 포함 요소

1. 검색
2. 필터
   - 전체
   - 작업 중
   - 기획서 있음
3. 페이지 리스트
   - 페이지명
   - 페이지 유형
   - 최근 작업 상태
4. 보조 작업 이력

### 페이지 아이템 표시 원칙

각 행에는 아래 정도만 표시한다.

- 페이지명
- 페이지 유형 / 그룹
- 최근 업데이트 또는 작업 상태

상태는 칩으로 너무 많이 붙이지 않고, 이번 단계 기준 3개만 고려한다.

- 정체성 있음/없음
- 요구사항 있음/없음
- 기획서 있음/없음

## 6. 우측 상세 구조

## 6.1 헤더

상단에는 아래만 둔다.

- 페이지명
- 페이지 유형 / 페이지 그룹
- 짧은 상태 요약

이 영역에서 builder, compare, version 액션을 크게 노출하지 않는다.

이번 단계는 문서 중심이기 때문이다.

## 6.1-a 상태 요약 바

헤더와 sticky anchor 사이에는 **상태 요약 바**를 둔다.

역할:

- 긴 상세 본문에 들어가기 전에 현재 페이지의 문서 진행 상태를 한 번에 보여준다
- 각 섹션의 완료 여부와 최신 상태를 빠르게 파악하게 한다

표시 항목:

1. 페이지 정체성
2. 요구사항
3. 기획서

각 항목은 아래 정도만 보여준다.

- 섹션명
- 상태 라벨
- 필요하면 짧은 보조 문구

예시:

- 정체성: 완료
- 요구사항: 저장됨
- 기획서: 업데이트 필요

중요:

- 이 바는 CTA 모음이 아니다
- “현재 문서 상태 요약” 역할만 한다

### 공통 상태 모델

세 섹션은 가능한 한 같은 상태 모델을 공유한다.

기본 상태:

- `empty`
- `draft`
- `ready`
- `stale`

해석:

- `empty`: 아직 작성된 내용 없음
- `draft`: 입력/수정 중이나 확정되지 않음
- `ready`: 현재 기준 문서로 사용 가능
- `stale`: 상위 입력이 바뀌어 다시 검토/생성이 필요함

### 섹션별 상태 예시

#### 페이지 정체성

- 없음 → `empty`
- 편집 중이지만 저장 전 → `draft`
- 저장 완료 → `ready`

#### 요구사항

- 비어 있음 → `empty`
- 저장 전 수정 있음 → `draft`
- 저장 완료 → `ready`

#### 기획서

- 아직 생성 안 됨 → `empty`
- 생성 중 / 직접 편집 중 → `draft`
- 최신 상태의 문서 → `ready`
- 요구사항 또는 정체성이 바뀐 뒤 재생성 필요 → `stale`

### stale 판정 규칙

기획서는 아래 경우 `stale`로 본다.

1. 기획서 생성 이후 요구사항이 저장됨
2. 기획서 생성 이후 페이지 정체성이 저장됨
3. 수동 편집된 기획서가 현재 요구사항과 명시적으로 어긋난다고 표시된 경우

즉 stale은 “문서가 존재하지만 최신 기준과 맞지 않을 수 있음”을 뜻한다.

## 6.2 sticky anchor

헤더 아래 고정 앵커 바를 둔다.

앵커 항목:

1. 페이지 정체성
2. 요구사항
3. 기획서

역할:

- 긴 상세 페이지 내 현재 위치 표시
- 클릭 시 해당 섹션으로 스크롤 이동
- 스크롤 위치에 따라 active 상태 변화

추가 역할:

- 각 섹션의 상태를 함께 보여준다
- 사용자가 본문을 스크롤하지 않아도 어느 섹션이 비어 있는지 파악하게 한다

중요:

- 이건 화면 전환 탭이 아니다
- 현재 상세 내부 목차다

### 앵커 상태 표현 규칙

앵커는 텍스트만 두지 않고, 각 섹션 상태를 작게 같이 표시한다.

표현 방식 예:

- 회색 점: `empty`
- 파란 점: `draft`
- 초록 점: `ready`
- 주황 점: `stale`

원칙:

- 점 또는 작은 배지 수준으로 최소 표현
- active 상태와 섞여서 복잡해 보이지 않게
- 현재 위치 표시보다 상태 표시가 더 강하게 보이면 안 된다

## 7. 각 섹션의 역할

## 7.1 페이지 정체성

이 섹션은 “기준 문서”다.

현재는 모달로 열리는데, 이번 구조에서는 상세 본문 첫 번째 섹션으로 승격해야 한다.

### 보여줄 항목

- 페이지 역할
- 페이지 목적
- 디자인 의도
- 반드시 유지할 것
- 피해야 할 것
- 시각 가드레일

### 원칙

- 설정 모달처럼 보이지 않게
- 이 페이지의 원래 목적을 설명하는 기준 문서처럼 보여야 함
- planner 실행 전에 먼저 읽는 영역처럼 구성

### 편집 모드 전환 방식

기본은 **읽기 보기**다.

- 정체성이 있으면: 항목을 카드 형태로 표시, 우측 상단에 “편집” 버튼
- “편집” 클릭 시: 폼 인라인 전환 (모달 불필요)
- 저장 방식: “저장” 버튼 클릭 시 즉시 저장 후 읽기 보기로 복귀
- 자동저장은 사용하지 않는다. 정체성은 기준 문서이므로 명시적 확정이 필요하다

### 빈 상태

정체성이 없는 페이지는 다음과 같이 표시한다.

- 항목 자리에 빈 카드 + 안내 문구: “아직 이 페이지의 정체성이 정의되지 않았습니다”
- “정체성 작성하기” 버튼으로 바로 편집 모드 진입
- 이 상태에서는 sticky anchor에 “정체성” 항목에 미완료 표시(점 또는 회색)를 추가한다

## 7.2 요구사항

이 섹션은 “이번 작업 입력”이다.

### 보여줄 항목

- 요청 제목
- 요청 배경
- 핵심 메시지
- 원하는 방향
- 피해야 할 방향
- 레퍼런스 URL
- 범위 선택
  - 페이지 전체
  - 특정 컴포넌트

### 원칙

- 여기서는 기획서 편집을 하지 않는다
- 사용자가 “이번 요청이 무엇인지”만 입력한다
- CTA는 `기획서 생성`

### 편집 모드 전환 방식

요구사항은 항상 편집 가능한 폼 상태로 열린다.

- 저장 방식: “저장” 버튼과 “기획서 생성” 버튼을 분리한다
  - “저장”: 요구사항만 저장, 기획서는 생성하지 않음
  - “기획서 생성”: 저장 후 즉시 기획서 생성 요청으로 이어짐
- 이전에 저장된 요구사항이 있으면 해당 내용이 폼에 채워진 상태로 열린다

### CTA 상태별 처리

| 상태 | CTA 라벨 | 동작 |
|------|----------|------|
| 기획서 없음 | 기획서 생성 | 저장 후 생성 시작 |
| 기획서 있음 | 기획서 재생성 | 기존 기획서 대체 경고 후 생성 |
| 생성 중 | 생성 중… | 비활성화, 진행 상태 표시 |
| 생성 실패 | 다시 시도 | 에러 메시지와 함께 재시도 가능 |

### 생성 중 상태 표시

기획서 생성이 진행되는 동안:

- 요구사항 섹션 하단에 진행 표시 바 또는 스피너 노출
- “기획서를 생성하고 있습니다. 잠시 기다려 주세요.” 안내 문구 표시
- 완료 시 기획서 섹션으로 자동 스크롤

### 빈 상태

요구사항이 없는 페이지는 다음과 같이 표시한다.

- 각 입력 항목이 비어 있는 폼 형태로 노출
- 상단 안내: “이번 작업에서 원하는 것을 입력하세요. 이 내용을 바탕으로 기획서를 생성합니다.”
- CTA는 “기획서 생성” (비활성화 상태 없이 항상 활성)

## 7.3 기획서

이 섹션은 고객용 PRD 문서다.

### 문서 구조

1. 한 줄 요약
2. 배경 / 문제 정의
3. 목표 범위
4. 핵심 기능
5. 사용자 흐름
6. 단계별 로드맵
7. 미결 사항 / 제외 판단 이유

### 원칙

- 기본은 렌더된 문서 보기
- 원문은 보조 액션으로만 노출
- builder용 내부 필드가 전면에 보이지 않게 한다

### 편집 방식

기획서는 LLM이 생성하는 것이 기본이다. 직접 편집은 보조 수단이다.

- 기본: LLM 생성 결과를 마크다운 렌더 뷰로 표시
- 수동 편집이 필요한 경우: “직접 편집” 버튼 → 마크다운 textarea 인라인 전환
- 수동 편집 후 저장하면 “직접 수정됨” 배지 표시 (LLM 재생성으로 덮어쓸 수 있음을 인지)
- LLM 재생성 시 직접 수정 내용 대체 경고를 요구사항 섹션의 CTA 단계에서 표시

### 빈 상태

기획서가 없는 페이지는 다음과 같이 표시한다.

- 빈 문서 자리에 안내 박스 노출
- 안내: “아직 기획서가 없습니다. 위 요구사항을 입력한 뒤 '기획서 생성'을 눌러 주세요.”
- 요구사항 섹션으로 스크롤하는 링크 포함

### 고객 전달 방식

기획서는 고객용 문서이므로 전달 수단이 필요하다.

- 우측 상단에 “내보내기” 버튼 노출
- 지원 형식: **PDF 다운로드**, **마크다운 복사**
- 공유 링크(읽기 전용 URL)는 이번 단계에서는 구현하지 않는다. 추후 workspace 기능과 함께 고려한다.

## 8. 표준 사용자 흐름

이 화면에서 사용자가 취하는 전형적인 흐름은 아래와 같다.

이 흐름을 기준으로 각 섹션의 순서와 CTA 위치를 판단한다.

### 신규 페이지 (정체성/요구사항/기획서 모두 없음)

```
1. 좌측 목록에서 페이지 선택
2. 우측 상세 진입 → 정체성 섹션에 빈 상태 안내 노출
3. "정체성 작성하기" 클릭 → 정체성 폼 인라인 전환
4. 항목 입력 후 "저장"
5. 요구사항 섹션으로 스크롤 (또는 앵커 클릭)
6. 이번 작업 요구사항 입력 후 "기획서 생성"
7. 생성 중 진행 표시 노출
8. 완료 시 기획서 섹션으로 자동 스크롤 → 렌더된 문서 확인
9. 필요 시 "내보내기"로 PDF 또는 마크다운 저장
```

### 반복 작업 (정체성 있음, 요구사항 업데이트)

```
1. 좌측 목록에서 페이지 선택
2. 정체성 섹션은 읽기 보기로 빠르게 확인
3. 요구사항 섹션으로 이동 → 이전 입력 내용이 채워진 폼 확인
4. 내용 수정 후 "기획서 재생성" 클릭
5. 기존 기획서 대체 경고 확인 → 생성 진행
6. 완료 후 기획서 섹션에서 결과 확인
```

### 기획서만 수정하는 경우

```
1. 기획서 섹션으로 직접 이동 (앵커 클릭)
2. "직접 편집" 버튼 클릭 → textarea 전환
3. 내용 수정 후 저장 → "직접 수정됨" 배지 표시
```

## 9. 좌측 목록 상태 구조

좌측 목록은 “페이지 선택용”이지만, 현재 작업 진행 정도를 최소한으로 보여주는 것이 좋다.

### 상태 표시 원칙

페이지 아이템마다 아래 세 상태만 요약한다.

1. 정체성
2. 요구사항
3. 기획서

표시 방식:

- 3개 작은 상태 점
- 또는 3개 짧은 축약 칩

색 규칙은 상태 요약 바와 동일하게 맞춘다.

- 회색: `empty`
- 파랑: `draft`
- 초록: `ready`
- 주황: `stale`

### 목적

- 목록에서 “아직 정체성이 없는 페이지”
- “요구사항은 있는데 기획서가 없는 페이지”
- “기획서가 낡은 페이지”

를 빠르게 찾게 한다.

### 목록에서 하지 말아야 할 것

- builder / compare / runtime 상태까지 같이 보여주기
- 상태 칩을 5개 이상 붙이기
- 긴 설명문 넣기

이번 단계에서 목록은 오직 문서 3종 상태만 다룬다.

## 10. 빌더 및 운영 영역 처리

이번 단계에서는 builder와 운영 메타를 없애는 것이 아니라, **전면에서 뒤로 보내는 것**이 목적이다.

### 메인 3섹션 뒤로 내릴 항목

- 디자인 빌더
- 비교 링크
- 버전 저장 / 핀
- acceptance
- runtime
- artifact
- coverage

### 처리 방식

- 기본 접힘
- “운영 도구” 섹션으로 묶기
- 문서 영역과 같은 위계로 보이지 않게 하기

### 레이어 원칙

운영 도구는 가능하면 **문서 레이어와 다른 계층**으로 보이게 해야 한다.

우선순위:

1. 별도 drawer / 분리 영역
2. 불가능하면 하단 접힘 섹션
3. 어떤 경우든 메인 3섹션과 같은 카드/헤더/여백 규칙을 쓰지 않음

즉 운영 도구는 같은 화면 아래에 있더라도,

- 배경 톤
- 제목 톤
- 여백 구조
- 액션 강조도

가 메인 문서 섹션과 분명히 달라야 한다.

### 운영 도구 진입 원칙

- 기본 진입 시 펼쳐져 있지 않아야 함
- 문서 작업 흐름을 방해하지 않아야 함
- “운영 도구 보기” 같은 명시적 액션 뒤에만 열려야 함

## 11. 상단 앵커 탭 도입 판단

상단 앵커 탭은 이번 구조에서 유효하다.

### 필요한 이유

- 우측 상세가 길다
- 세 섹션을 오가며 읽어야 한다
- 좌측은 페이지 선택용이라 상세 내부 내비가 부족하다

### 주의점

- 글로벌 탭처럼 보이면 안 된다
- “현재 상세 내부 목차”처럼 보여야 한다
- 섹션 제목과 앵커 라벨은 동일해야 한다

## 12. 추천 구현 순서

각 단계는 독립적으로 배포 가능하다. 이전 단계가 완료되지 않아도 다음 단계를 시작할 수 있다.

### 1단계 — 페이지 정체성 본문 승격

- 페이지 정체성 모달을 본문 첫 번째 섹션으로 이동
- 읽기 보기 / 편집 보기 전환 구현
- 빈 상태 안내 추가

완료 기준: 정체성 내용을 모달 없이 본문에서 읽고 편집할 수 있다

### 2단계 — 요구사항 / 기획서 섹션 분리

- 요구사항 폼과 기획서 렌더 뷰를 화면상 완전히 분리
- 요구사항 저장 버튼과 기획서 생성 CTA 분리
- 기획서 생성 중 로딩 상태 표시 구현
- 기획서 직접 편집 기능 추가

완료 기준: 요구사항 입력 → 기획서 생성 → 결과 확인 흐름이 한 화면에서 막힘 없이 동작한다

### 3단계 — sticky anchor 추가

- 우측 상세 상단에 고정 앵커 바 추가
- 스크롤 위치에 따른 active 상태 전환 구현
- 각 앵커 항목에 완료/미완료 상태 표시 추가

완료 기준: 긴 상세 화면에서 앵커 클릭으로 세 섹션 간 이동이 즉시 가능하다

### 4단계 — 운영 도구 하단 격리

- builder/compare/version/runtime 영역을 “운영 도구” 섹션으로 묶기
- 기본 접힘 처리
- 문서 3섹션과 시각적으로 분리된 영역으로 표현

완료 기준: 화면 진입 시 운영 도구가 기본 접힘 상태이며, 메인 3섹션이 전면에 명확히 보인다

### 5단계 — 좌측 목록 상태 중심 정리

- 검색 + 상태 필터 (정체성/요구사항/기획서 있음/없음) 구현
- 각 페이지 아이템에 3가지 상태 칩 표시
- 기존 상태 칩 정리 (이번 단계와 무관한 칩 제거)

완료 기준: 목록에서 “기획서 없음” 필터로 미작업 페이지만 빠르게 추릴 수 있다

### 6단계 — 기획서 내보내기

- 기획서 섹션에 “내보내기” 버튼 추가
- PDF 다운로드 구현
- 마크다운 클립보드 복사 구현

완료 기준: 기획서를 PDF 파일로 저장하거나 마크다운 텍스트로 복사할 수 있다

## 13. 결론

이번 단계에서 좋은 관리자 화면은 “기능이 많은 화면”이 아니다.

오히려 아래가 명확한 화면이다.

- 좌측은 페이지를 고른다
- 우측은 정체성/요구사항/기획서를 읽고 쓴다
- 현재 위치는 상단 앵커로 잡는다
- 빌더와 운영 메타는 뒤로 뺀다

즉 이번 관리자 화면의 핵심은

**문서 중심 상세 화면으로 재구성하는 것**

이다.
