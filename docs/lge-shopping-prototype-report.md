# LG전자 쇼핑몰 프로토타입 전환 검토 보고서

## 1. 목적

본 과제는 `lge.co.kr` 단일 사이트를 기준 자산으로 삼아, 사이트 전체 구조를 분석하고 이를 재사용 가능한 컴포넌트 자산으로 전환한 뒤, 이후 텍스트 또는 Figma Page 단위 변경 요구를 반영할 수 있는 운영형 프로토타입 체계를 구축하는 데 목적이 있습니다.

핵심 목표는 신규 사이트를 범용적으로 자동 생성하는 것이 아니라, 기존 LG전자 쇼핑몰을 기준으로 `설명`, `수정`, `재조합`이 가능한 웹 프로토타입 운영 기반을 만드는 것입니다. 또한 상위자 보고 및 고객 시연 목적상, 초기 프로토타입 단계에서는 디자인 표현을 최대한 원본에 가깝게 보존하는 것을 우선 원칙으로 둡니다.

## 2. 분석 결과 요약

`lge.co.kr`의 `robots.txt`와 공개 `sitemap.xml`을 기준으로 확인한 결과, 사이트는 다수의 하위 사이트맵으로 분리되어 있으며 주요 공개 구조는 아래와 같습니다.

- 홈: `https://www.lge.co.kr/home`
- 카테고리 리스트: `https://www.lge.co.kr/category/...`
- 제품 상세: 개별 제품 상세 URL
- 구독/케어: `https://www.lge.co.kr/care-solutions`
- 브랜드: `lg-signature`, `objet-collection`, `lg-thinq`
- 고객지원: `https://www.lge.co.kr/support`
- 베스트샵: `https://www.lge.co.kr/bestshop`

확인된 사이트맵 기준 수량은 대략 아래 수준입니다.

- 제품 리스트 페이지: 314개
- 제품 상세 페이지: 46,031개
- 구독/케어 페이지: 66개
- 고객지원 페이지: 25개

이 규모를 고려하면 전체 사이트를 한 번에 정밀 구현하는 접근은 비효율적입니다. 다만 이번 과제는 단일 사이트만을 대상으로 하므로, 범위를 넓히는 대신 깊이를 확보하는 방식이 적절합니다. 즉, 공개 가능한 페이지를 최대한 깊게 수집하고, 이미지 자산과 팝업 상태까지 포함하여 시각적으로 원본에 가까운 프로토타입 기반을 확보한 뒤, 이를 컴포넌트 체계로 정규화하는 방식으로 접근해야 합니다.

## 3. 추진 방식

본 과제는 아래 4단계로 진행하는 것이 타당합니다.

### 3.1 1단계: 사이트 전체 구조 수집 및 분석

첫 단계에서는 대상 사이트의 URL 구조, 페이지군, 섹션 패턴, 반복 UI를 수집합니다. 이 단계의 기준은 단순 구조 확인이 아니라, 상위자가 원본과의 유사성을 즉시 판단할 수 있을 정도로 시각 자산과 상태를 충분히 확보하는 것입니다.

- 사이트맵 기반 URL 목록 확보
- 홈/카테고리/상세/구독/지원/브랜드 페이지군 분류
- 데스크톱/모바일 기준 주요 레이아웃 차이 수집
- 반복되는 블록 구조 식별
- 페이지 내 이미지, 배너, 아이콘 등 주요 시각 자산 폴링
- 버튼 클릭, 탭, 드롭다운, 모달, 팝업 등 숨은 상태 확인
- 링크 이동 가능한 깊은 페이지까지 추적
- 크롤링 제한 페이지는 공통 대체 페이지로 분류

이 단계는 시간이 가장 오래 걸리지만 이후 전 과정의 기준 데이터가 됩니다.

### 3.2 2단계: 컴포넌트화 및 페이지 묶음 구조화

분석 결과를 HTML 단위가 아니라 `컴포넌트`와 `페이지 구조`로 재정리합니다. 다만 이번 과제에서는 시각 표현도 중요하므로, 단순 구조 추상화가 아니라 원본 디자인과 연결된 자산 단위로 묶어야 합니다.

- 공통 UI를 컴포넌트 단위로 추출
- 페이지를 섹션 조합 구조로 저장
- 상위 페이지별로 묶음 구조 생성
- 신규/기존 컴포넌트를 분리 관리
- 이미지/배경/배너 자산과 컴포넌트를 연결 저장
- 팝업/모달/오버레이를 일반 섹션이 아닌 상태성 컴포넌트로 분리

예상 공통 자산은 아래와 같습니다.

- Header / Footer
- Hero / Promotion Banner
- Category Navigation
- Product Card Grid
- Filter / Sort Panel
- Product Summary Block
- Price / Benefit Block
- Review / Recommendation Block
- CTA Banner
- Support Shortcut Block

### 3.3 3단계: 텍스트 또는 Figma Page 기반 변경 계획 수립

기존 사이트를 컴포넌트 구조로 정리한 뒤에는, 변경 요구를 `신규 생성`이 아니라 `기존 구조 위 변경 계획`으로 처리합니다.

- 텍스트 입력: 문구, 순서, 섹션 추가/삭제, CTA 수정
- Figma Page 입력: 신규 섹션 후보, 디자인 대체안, 페이지 재배치안

즉, 변경 입력은 화면을 다시 처음부터 만드는 것이 아니라, 현재 자산 체계 위에 어떤 컴포넌트를 추가/삭제/교체할지 결정하는 용도로 사용합니다.

### 3.4 4단계: 컴포넌트 자산화 및 상시 재조합 운영

분석되거나 새로 정의된 내용은 모두 컴포넌트 자산으로 편입합니다.

- 공통 컴포넌트 레지스트리 저장
- 페이지별 사용 이력 및 조합 구조 저장
- 이후 페이지별 재조합 가능 상태 유지
- 추가/삭제/순서 변경 가능한 운영 UI 제공

이 구조가 완성되면 운영자는 페이지를 다시 제작하지 않고, 보유 컴포넌트를 조합하여 원하는 프로토타입을 즉시 재구성할 수 있습니다.

## 4. 핵심 설계 원칙

본 과제는 `복제`가 아니라 `컴포넌트화 후 재구성`을 원칙으로 합니다. 다만 초기 보고 및 시연 품질 확보를 위해, 수집 단계에서는 가능한 범위 내에서 원본 디자인을 최대한 보존합니다.

- URL은 원본 HTML 복제가 아니라 구조 분석 후 컴포넌트 매핑
- Figma는 레이어 복제가 아니라 웹 컴포넌트 구조로 해석
- 변경 요청은 전체 재생성이 아니라 부분 교체 및 부분 추가
- 시각 자산은 가능한 한 원본 이미지와 배경 리소스를 재사용
- 팝업과 모달은 별도 상태로 캡처 후 컴포넌트 라이브러리에 포함
- 크롤링 제한 페이지는 공통 플레이스홀더 페이지로 처리

이 원칙을 채택해야 수정 속도, 재사용성, 운영 안정성을 확보할 수 있습니다.

## 5. 구현 방향

현재 기준으로 가장 적절한 구조는 다음과 같습니다.

- 수집 계층: 사이트맵 수집, 페이지 렌더 결과 저장, desktop/mobile 구조 수집, 이미지/배너 자산 폴링, 팝업 상태 수집
- 정규화 계층: page / section / component / props / interaction / responsive 형태로 변환
- 운영 계층: 페이지 미리보기, 컴포넌트 추가/삭제, 순서 변경, 표시 여부 제어
- 변경 계층: 텍스트 또는 Figma Page 기준 변경 계획 반영
- 오케스트레이션 계층: LLM이 분석, 매핑, 변경 계획, 영향 범위를 보조

LLM은 직접 화면을 자유 생성하는 용도가 아니라, `사이트 구조 해석`, `컴포넌트 매핑`, `변경 계획 수립`, `부분 수정 지시` 역할로 두는 것이 적절합니다.

특히 이번 과제의 수집 단계에서는 아래 4개를 필수 정책으로 둡니다.

1. 가능한 링크는 깊은 페이지까지 추적한다.
2. 원본 이미지와 배너 리소스는 별도 자산으로 저장한다.
3. 팝업, 모달, 드롭다운 상태는 클릭 기반으로 확인 후 별도 캡처한다.
4. 크롤링 제한 경로는 공통 페이지 템플릿으로 대체하여 흐름상 공백이 생기지 않게 한다.

## 6. 결론

본 과제는 LG전자 쇼핑몰을 대상으로 아래 흐름으로 추진하는 것이 타당합니다.

1. 사이트 전체 구조를 수집하고 페이지군을 정리한다.
2. 이미지, 배너, 팝업 상태를 포함한 수집 결과를 컴포넌트 및 페이지 구조로 전환한다.
3. 텍스트 또는 Figma Page 기반 변경 계획을 이 구조 위에 반영한다.
4. 새로 분석되거나 생성된 결과는 모두 컴포넌트 자산으로 편입한다.

이 방식은 단발성 시안 제작이 아니라, `기존 사이트를 기반으로 한 설명·수정·확장 가능한 프로토타입 운영 체계`를 구축하는 방향으로 적절합니다.
